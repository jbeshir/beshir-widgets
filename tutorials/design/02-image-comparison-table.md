# Designing the Image Comparison Table

The Image Comparison Table is a grid widget that puts multiple AI models side by side — themes run down the left, models span the top, and each intersection is a thumbnail generated by that model for that theme. Try it live at https://image-comparison-table.widgets.beshir.org . The JavaScript half — table parsing and state, the fit-to-viewport ResizeObserver, and the lightbox keyboard handler — is covered in the sibling tutorial [../frontend/02-image-comparison-table.md](../frontend/02-image-comparison-table.md). This is entry 02 of the `design/` track and assumes familiarity with [tutorial 01](./01-function-plotter.md); every CSS technique introduced here is listed in the closing ledger and may be referenced by name in later tutorials without re-explanation.

## The look

The widget is a clean comparison grid: theme labels down the left, model names across the top, square thumbnails filling each cell. Clicking any thumbnail opens a frosted-glass lightbox; an ⓘ button on each model header reveals the generation prompt in a floating popover. The table collapses to a stacked two-column layout automatically when the viewport narrows below 560 px.

The design language — slate background tokens, indigo accent, soft layered shadows — is the same house style established in [tutorial 01](./01-function-plotter.md), and is inherited here without re-description. What this tutorial adds is depth on the *structural* CSS: a Grid layout whose column count is data-driven, a full layout restructure triggered by a single class toggle, three layers of positioned overlays, CSS math functions for fluid sizing, keyframe animations for entrance effects, transform functions for hover lift and rotation, `backdrop-filter` for frosted-glass surfaces, generated content for decorative and typographic details, and a complete keyboard-interaction polish layer.

## CSS Grid: Building the Comparison Table

The widget's central challenge is alignment: however many AI models appear in the table, every row's cells must form true columns — the label for "Fantasy landscape" must sit in exactly the same horizontal slot as the label for "Photorealistic portrait," and each model's thumbnails must stack into a vertical column regardless of how many rows exist. That can't be done with flexbox alone, because flex containers are one row at a time and have no concept of cross-row alignment. What's needed is a single grid whose column tracks span every row simultaneously.

### The grid declaration

The `.grid` element establishes that shared column structure:

```css
/* styles.css:182–186 */
.grid {
  display: grid;
  grid-template-columns: minmax(120px, 0.7fr) repeat(var(--cols), minmax(0, 1fr));
  gap: 14px;
}
```

`display: grid` creates a block-level grid formatting context. Unlike block layout, grid items don't collapse margins with each other, and the container controls sizing in two dimensions simultaneously.

`grid-template-columns` defines the width of each column track. Two kinds of track appear here.

**The `fr` unit.** `fr` means "one fraction of the remaining free space" — the space left over after fixed track sizes and gaps are subtracted from the container width. `0.7fr` versus `1fr` means the label column takes 0.7 shares while each model column takes a full share, making the label narrower than any single data column. No pixel arithmetic needed: the browser divides the free space proportionally on every resize.

**`minmax(MIN, MAX)`.** Each track is sized with `minmax()`, which gives the browser a range: grow as large as MAX, but never shrink below MIN.

- `minmax(120px, 0.7fr)` — the label column has a hard floor of 120 px. Even if `0.7fr` resolves to something smaller on a cramped viewport, the track won't collapse below that width, keeping row labels legible.
- `minmax(0, 1fr)` — the model columns have a zero minimum. This is the anti-blowout idiom: without it, `1fr` secretly means `minmax(auto, 1fr)`, where `auto` resolves to the track's *min-content* size — the narrowest a grid item can be without overflowing its content. A single wide image or long filename can inflate that minimum and blow the whole grid beyond the container. Setting the minimum to `0` removes that floor: the track is allowed to shrink to nothing, and the `1fr` maximum is what actually governs its size in practice.

This is the same problem [tutorial 01](./01-function-plotter.md) solved with `min-width: 0` on the flex input: grid items, like flex items, default to `min-width: auto`, which resolves to min-content size. There, the fix was an explicit `min-width: 0` on the item itself; here, writing the track minimum as `0` in `minmax()` achieves the same thing at the container level. The `.grid-cell` rule provides a belt-and-suspenders guard at the item level too:

```css
/* styles.css:192–194 */
.grid-cell {
  min-width: 0;
}
```

Both together ensure no single cell's content can push a track beyond its allotted `fr`.

**`repeat(var(--cols), minmax(0, 1fr))`.** `repeat(N, track-definition)` expands a track pattern N times — rather than listing `minmax(0, 1fr)` four times for a four-model table, `repeat(4, minmax(0, 1fr))` does it in one declaration. Here the count isn't a literal integer; it's `var(--cols)`, a [custom property](./01-function-plotter.md) read from the element's style.

That property is set by `Grid.tsx` as an inline style:

```tsx
/* Grid.tsx:19 */
style={{ '--cols': cols } as any}
```

`cols` is `table.columns.length` — the data-driven column count from JavaScript. The inline style injects it directly onto the `.grid` element, where CSS sees it exactly as it would see any other custom property. When the browser evaluates `var(--cols)` inside `repeat()`, it substitutes the integer, and the `grid-template-columns` declaration expands to as many `minmax(0, 1fr)` tracks as there are model columns. Add a column to the data, and the grid gains a track automatically — no CSS change required.

**`gap: 14px`.** Gutter spacing between tracks, using the same `gap` shorthand [tutorial 01](./01-function-plotter.md) introduced in the flex context — `gap` works identically in grid.

### Flattening the rows with `display: contents`

The markup wraps each row's cells in a `.grid-row` div:

```tsx
/* Grid.tsx:21 */
<div class="grid-row grid-row--head" role="row">
```

Without intervention, `.grid-row` would be a direct child of `.grid` and therefore a *single grid item* — a block that occupies one cell. The cells inside it would be children of that block, not of the grid, so `grid-template-columns` would have no effect on their widths.

```css
/* styles.css:188–190 */
.grid-row {
  display: contents;
}
```

`display: contents` removes the element's own boxes from the layout tree — the `.grid-row` div generates neither an outer block box nor an inner formatting context. Its children are "lifted" directly into the parent formatting context, as if the wrapper weren't there. The result: every `.grid-cell` across every row becomes a direct grid item of `.grid`, and a single `grid-template-columns` declaration aligns all of them into columns.

> **Accessibility caveat.** `display: contents` has a multi-year history of also removing elements from the *accessibility* tree — meaning ARIA roles applied to the element can be silently dropped in some browsers. Safari has been the most affected engine; Chrome fixed the main case in 2021 but regressions have resurfaced. `Grid.tsx` sets `role="row"` on every `.grid-row` div to make the table structure machine-readable, but that role may not survive `display: contents` in all assistive-technology combinations. Test with real screen readers. The frontend tutorial [`../frontend/02-image-comparison-table.md`](../frontend/02-image-comparison-table.md) covers the ARIA table role structure and the current state of browser conformance in detail.

### A second flavour: intrinsic auto-fill

The widget's image picker uses an entirely different approach to grid column-counting:

```css
/* styles.css:771–773 */
display: grid;
grid-template-columns: repeat(auto-fill, minmax(258px, 1fr));
gap: 14px;
```

Instead of a data-driven integer, the repeat count is `auto-fill` — a browser keyword that means "create as many tracks as will fit in the container without exceeding it." The browser takes the minimum from `minmax()` (258 px), calculates how many 258 px tracks (plus gaps) can be packed into the available width, and creates exactly that many. As the viewport narrows below a threshold, a track is removed; as it widens, one is added. No JavaScript, no media queries.

`auto-fill` and `auto-fit` both do this, but they differ when there are *fewer items than tracks*. `auto-fill` keeps the empty tracks in place — the grid maintains consistent column widths regardless of how many cards it holds. `auto-fit` collapses empty tracks to zero, causing the filled tracks to stretch and take over the vacated space. For a picker where the card count is unknown and more items might load, `auto-fill` is the more predictable choice: each card stays a consistent 258 px–1fr width.

The comparison table can't use this approach — its column count must match the data exactly, which requires the JS-driven `var(--cols)` mechanism. But both grids use the same underlying tools: `repeat()`, `minmax()`, and `fr`. The only difference is whether the browser or the data decides the track count.

With the column skeleton in place, the next section shows how a single modifier class on a parent — `.grid-wrap--compact` — can dismantle that skeleton and rebuild it as a stacked single-column layout, without changing a single element in the DOM.

## Compact-Layout Restructure: One DOM, Two Grids

An N-column comparison table works fine in a full-width browser tab. Embedded in a blog post or a narrow sidebar it falls apart: the columns compress until the thumbnails are unreadable and the label column is squeezed into a sliver. The usual remedies — hiding columns, truncating data, reaching for a separate mobile template — each discard information or duplicate markup. This widget takes a different approach: the same DOM, the same cells, the same grid children, re-laid by a single class on the ancestor.

### The trigger

`App.tsx:99` makes the decision:

```tsx
const compact = width < 560;
```

A `ResizeObserver` feeds the widget's pixel width into this one comparison. When it flips, `Grid.tsx:14` adds or removes the modifier class:

```tsx
<div class={`grid-wrap${compact ? ' grid-wrap--compact' : ''}`}>
```

JS owns *when* the layout changes. CSS owns *how*. Everything below is CSS.

For the ResizeObserver wiring and how the `width` value is kept current, see the sibling frontend tutorial [`../frontend/02-image-comparison-table.md`](../frontend/02-image-comparison-table.md).

### Step 1: Collapse the outer grid

```css
/* styles.css:467–470 */
.grid-wrap--compact .grid {
  grid-template-columns: 1fr;
  gap: 0;
}
```

The base rule (§1) gives `.grid` a multi-column `grid-template-columns` driven by `var(--cols)`. This compact override replaces it with a single `1fr` track — one column, full width — and removes the gap. Specificity of `.grid-wrap--compact .grid` is **(0,2,0)** (two classes); the base `.grid` rule is **(0,1,0)**. Two classes beat one, so no `!important` and no source-order dependency is needed: the compact selector wins because it is intrinsically more specific. See [tutorial 01](./01-function-plotter.md) for the (A,B,C) accounting.

### Step 2: Undo `display: contents`

```css
/* styles.css:472 */
.grid-wrap--compact .grid-row { display: block; margin-bottom: 18px; }
```

Section §1 taught that `.grid-row { display: contents }` dissolves the row wrapper so its cells participate directly in the outer grid. In compact mode that transparent wrapper is a liability — we need each row to be a real box again so it can become its own grid. `display: block` re-establishes the box. Again, specificity **(0,2,0)** beats the base **(0,1,0)**: pure cascade, no tricks.

### Step 3: Hide the column header row

```css
/* styles.css:473 */
.grid-wrap--compact .grid-row--head { display: none; }
```

The header row (the one with model names) made sense when every column had a heading above it. With each data row now a self-contained layout, the column headers have no visual home — and repeating them for every row would be noise. `display: none` removes the element from both the box tree and the accessibility tree. Because the widget already handles accessible labelling via `role="columnheader"` on those cells, hiding them in compact view is safe: column identity is instead surfaced per-thumbnail as a styled label badge (see the [`attr()` section](#attr--reading-attribute-values-into-content) below).

### Step 4: Each data row becomes its own two-column grid

```css
/* styles.css:480–485 */
.grid-wrap--compact .grid-row:not(.grid-row--head) {
  display: grid;
  grid-template-columns: 1fr 1fr;
  gap: 8px;
  align-items: stretch;
}
```

This is where the layout flips from "one shared grid" to "one small grid per row." Each data row becomes a two-column grid; within it, the label spans both columns (next step) and the image cells sit side by side.

The `:not(.grid-row--head)` pseudo-class targets every `.grid-row` that is *not* the header. `:not()` accepts a selector as its argument and matches elements that fail that selector. Crucially, `:not()` contributes the **specificity of its argument** — not zero, not its own. Here the argument is `.grid-row--head`, a single class at **(0,1,0)**, so `:not(.grid-row--head)` adds **(0,1,0)** to the compound.

Full specificity of `.grid-wrap--compact .grid-row:not(.grid-row--head)`:

| Part | (A,B,C) |
|---|---|
| `.grid-wrap--compact` | (0,1,0) |
| `.grid-row` | (0,1,0) |
| `:not(.grid-row--head)` | (0,1,0) |
| **Total** | **(0,3,0)** |

The base rule targeting the same elements (`.grid-row`) sits at **(0,1,0)**. The compact selector wins by three class-weight columns to one, without touching source order.

### Step 5: Span the label across both columns

```css
/* styles.css:486 */
.grid-wrap--compact .grid-cell--label { grid-column: 1 / -1; }
```

`grid-column` is shorthand for `grid-column-start / grid-column-end`. Both values are **grid line numbers**; lines count from `1` at the start edge. The value `-1` always refers to the **last line of the explicit grid** — negative indices count from the end. With `grid-template-columns: 1fr 1fr`, the explicit grid has three lines (before track 1, between tracks, after track 2), so `-1` is line 3. The declaration is equivalent to `grid-column: 1 / 3`, but `-1` is more robust: if the column count ever changes, the label still spans everything without the rule needing to change.

The result: the label row sits above the two image cells, behaving like a section header for each theme prompt.

### Step 6: Suppress the accent bar

```css
/* styles.css:487 */
.grid-wrap--compact .grid-cell--label::before { display: none; }
```

The label cell has a decorative `::before` pseudo-element — a 3 px coloured accent bar (described fully in §8). In compact mode the label already sits in its own row with a bottom border; the accent bar would double up the visual separator. `display: none` suppresses it.

### How the two layers interact

The base rules (§1) and the compact rules target the same elements. When `.grid-wrap--compact` is absent, only the base rules are active — the more-specific compact selectors simply never match. When the class is present, every compact selector **outranks** its base counterpart because it carries at least one extra class. The entire layout restructure is pure cascade: no `!important`, no inline overrides, no JavaScript style mutations beyond the one class toggle. The cascade is doing exactly what it was designed to do.

With the two-dimensional layout and its responsive restructure covered, the next section turns to depth: how absolutely-positioned overlays and `z-index` layer the zoom badge, the info popover, and the lightbox on top of the grid.

## Positioning & Stacking Contexts

Three layers of UI float above the comparison grid: a zoom badge clipped to each thumbnail's corner, a popover anchored below its trigger button, and a full-viewport lightbox modal. Each is a positioned element, and the relationship between them — which one paints over which — is governed by stacking contexts. Understanding how those work is the difference between positioning being a power tool and a source of undiagnosable bugs.

### Containing blocks: `position: relative`

The zoom badge (`.thumb-zoom`) lives inside `.grid-cell--img`. To pin it to the thumbnail's corner, the badge needs `position: absolute` — but absolute elements position themselves against their nearest *positioned* ancestor (one whose `position` is not `static`). Without intervention, that ancestor would be whatever happens to be positioned further up the tree, likely far removed from the thumbnail cell.

The fix is one declaration on the parent:

```css
/* styles.css:259–262 */
.grid-cell--img {
  position: relative;
  display: block;
  width: 100%;
```

`position: relative` keeps `.grid-cell--img` exactly where normal flow placed it — it doesn't move a pixel — but it promotes the element to *positioned* status, establishing a containing block for any absolutely positioned child. The badge now has a reliable anchor.

The same pattern repeats for the info popover. `.info-wrap` wraps both the trigger button and the floating panel:

```css
/* styles.css:341–345 */
.info-wrap {
  position: relative;
  display: inline-flex;
  align-items: center;
}
```

Again, `relative` here isn't about offset; it's about establishing jurisdiction.

### `position: absolute` — removed from flow

With containing blocks in place, the children go absolute. The zoom badge is pinned to the thumbnail's top-right corner:

```css
/* styles.css:307–310 */
.thumb-zoom {
  position: absolute;
  top: 8px;
  right: 8px;
```

`top` and `right` resolve against the containing block's edge. The badge sits 8 px from each corner regardless of cell width — a fixed mount. Notice `pointer-events: none` at `styles.css:323`: the badge overlaps the clickable button underneath, and without this declaration it would intercept the click intended to open the lightbox. With it, the badge is visually present but transparent to pointer events.

The info popover is more involved. It needs to float directly below its trigger and be horizontally centred over it — at whatever computed width the popover happens to be:

```css
/* styles.css:378–382 */
.info-popover {
  position: absolute;
  top: calc(100% + 10px);
  left: 50%;
  transform: translateX(-50%);
```

`top: calc(100% + 10px)` places the popover's top edge just past `.info-wrap`'s bottom edge (`100%` of the containing block's height), then adds 10 px of clearance. `calc()` is the tool that makes this expressible as a single value — it mixes `%` and `px` in one arithmetic expression where neither unit alone would work. The CSS Math section (§4) builds on this foundation.

The centering idiom that follows is worth memorising: `left: 50%` positions the popover's *left edge* at the midpoint of the containing block. That overshoots — the popover extends rightward past centre. `transform: translateX(-50%)` corrects it by nudging the element left by half its *own* width. Because `translateX` percentages resolve against the element itself rather than the containing block, the expression centres the popover regardless of its computed width. The lightbox uses the vertical variant: `top: 50%; transform: translateY(-50%)` on `.lb-nav` (`styles.css:597–598`) keeps the navigation arrows pinned to the midpoint of the image area at any height.

The decorative arrow above the popover applies both axes at once:

```css
/* styles.css:405–415 */
.info-arrow {
  position: absolute;
  top: -7px;
  left: 50%;
  ...
  transform: translateX(-50%) rotate(45deg);
}
```

`translateX(-50%)` centres a 12×12 px square over the popover's midpoint; `rotate(45deg)` then tilts it 45° to produce the diamond arrow shape. This is a real DOM element (`<span class="info-arrow" aria-hidden="true" />` in `InfoPopover.tsx`), not a pseudo-element — the transform section (§6) covers why function order in a chained `transform` matters.

### `position: fixed` — viewport anchor

The lightbox backdrop must cover the entire viewport and stay in place as the user scrolls. `position: absolute` would scroll away with the page. `position: fixed` positions the element against the viewport itself:

```css
/* styles.css:519–540 */
.lb-backdrop {
  /* Native <dialog>: override UA defaults ... */
  position: fixed;
  inset: 0;
  width: 100%;
  height: 100%;
  max-width: 100%;
  max-height: 100%;
  margin: 0;
  border: 0;
  ...
  z-index: 100;
  ...
  animation: lb-fade 140ms ease-out;
}
```

`inset: 0` is the four-side shorthand — equivalent to `top: 0; right: 0; bottom: 0; left: 0` — that stretches the backdrop across the entire viewport. Fixed elements stay at those coordinates regardless of scroll position; the document moves behind them.

`.lb-backdrop` is now a native `<dialog>` element opened via `showModal()`, which places it in the browser's **top layer** — a rendering surface above every stacking context on the page regardless of `z-index` (the `z-index: 100` is now belt-and-braces). The UA centres dialogs by default with `margin: auto` and an `auto` inset; the `width: 100%; height: 100%; max-width: 100%; max-height: 100%; margin: 0; border: 0` declarations neutralise those defaults so the dialog fills the viewport exactly as the former `<div>` backdrop did. A companion rule at `styles.css:544–546` keeps the UA-generated `::backdrop` transparent — the frosted dim lives on the `<dialog>` element itself and must not double up behind it. For the focus-trap, `cancel`/`close` event wiring, and accessibility detail, see [`../frontend/02-image-comparison-table.md`](../frontend/02-image-comparison-table.md) ("Modal dialog" section).

### Stacking contexts — the heart of it

With positioned elements at multiple levels, something has to decide which one paints last (and therefore on top). That something is the stacking context.

A **stacking context** is an isolated z-axis group. `z-index` values only compare *within the same stacking context*. A stacking context is created by:

- `position: fixed` or `position: sticky` (unconditionally)
- A positioned element with `z-index` other than `auto`
- Any element with `transform ≠ none`, `opacity < 1`, `filter ≠ none`, or `backdrop-filter ≠ none`

The root element creates the first context. The popover (`z-index: 80`, `styles.css:390`) is a direct child of the root context — any other root-level stacking context with a higher index paints over it. The lightbox backdrop (`z-index: 100`, `styles.css:537`) is now a native `<dialog>` in the browser's top layer via `showModal()`, which paints above every stacking context on the page regardless of `z-index`; the `z-index: 100` is retained as belt-and-braces.

The constraint that trips people up: a z-index on a child cannot escape its parent's stacking context. If `.grid-cell--img` permanently carried `transform ≠ none`, it would form its own stacking context, trapping the zoom badge inside it. A badge with `z-index: 9999` inside that context would still sit *below* any root-level sibling with `z-index: 2`. The widget avoids this by keeping the card's hover lift — `transform: translateY(-3px)` at `styles.css:277` — as a transient `:hover` state, not a permanent property, and by placing the popover in `.info-wrap`, not inside the thumbnail.

`transform` creating a stacking context is a recurring theme: it comes up again in §6 and §7, which also cover how a transformed ancestor can silently break `position: fixed` descendants by becoming their containing block instead of the viewport.

The offsets in this section — `calc(100% + 10px)`, `min(360px, 80vw)`, `min(420px, 92vw)` on the popover's width constraints — all rely on CSS math functions to express responsive geometry in a single declaration; the next section examines `clamp()`, `min()`, and `max()` in full.

## CSS Math Functions: clamp(), min(), max()

Section §3 introduced `calc()` as the tool for mixing units in a single value — the popover sits at `top: calc(100% + 10px)` (`styles.css:380`) because no other mechanism can add a pixel offset to a percentage. Three functions build on that foundation: `clamp()`, `min()`, and `max()`. All four resolve at computed-value time, after `var()` substitution and before layout, so the browser can freely mix `vw`, `px`, `%`, and `vh` in a single expression. The payoff is responsive sizing that scales continuously rather than jumping at discrete breakpoints — the approach [tutorial 01](./01-function-plotter.md) used `@media` for was the right tool for discrete layout switches, but for values that should track the viewport on a smooth curve, a math function does it in one declaration with no query at all.

### Fluid type with clamp()

```css
/* styles.css:140 */
font-size: clamp(19px, 1.6vw + 12px, 24px);
```

`clamp(MIN, PREFERRED, MAX)` takes three arguments. The browser evaluates the preferred expression — here `1.6vw + 12px` — and then clamps it to the range `[MIN, MAX]`. It is exactly equivalent to `max(MIN, min(PREFERRED, MAX))`, which is why the spec calls the three functions a family.

Walk through a few widths to see the tracking and pinning in action:

- **375 px viewport** (a common phone): `1.6vw + 12px = 1.6 × 3.75 + 12 = 18px`. That falls below the 19 px floor, so the font resolves to **19 px**.
- **800 px viewport**: `1.6 × 8 + 12 = 24.8px`. That exceeds the 24 px ceiling, so the font pins to **24 px**.
- **500 px viewport**: `1.6 × 5 + 12 = 20px`. That sits comfortably between 19 and 24, so the font tracks at **20 px**.

Between roughly 437 px and 750 px the heading scales continuously with the viewport. Below 437 px and above 750 px it is pinned. No `@media` block, no duplicated rule, no risk of a cascade collision — the entire responsive behaviour lives on one line.

### min() as a viewport-aware ceiling

`min(A, B)` picks the **smaller** of its arguments. The mental model is: it sets a ceiling that can be lowered further by a second constraint.

The info popover uses it on both of its width bounds:

```css
/* styles.css:383–384 */
min-width: min(360px, 80vw);
max-width: min(420px, 92vw);
```

On a 1200 px desktop, `80vw = 960px` and `92vw = 1104px` — both larger than 360 and 420, so the fixed px values win and the popover is constrained between 360 and 420 px. On a 340 px phone, `80vw = 272px` and `92vw = 313px` — both smaller than the px values, so the vw values win and the popover shrinks to fit the screen. Without `min()`, a fixed `min-width: 360px` on a 340 px viewport would force horizontal overflow; with it, the popover gracefully narrows. The viewport formula is the safety valve; the pixel value is the comfortable maximum.

### Composing min() and calc()

The lightbox image shows the natural next step: nesting `calc()` inside `min()`.

```css
/* styles.css:589 */
max-height: min(760px, calc(100vh - 180px));
```

`calc(100vh - 180px)` reserves space for browser chrome and the lightbox caption — 180 px of vertical headroom. `min(760px, calc(100vh - 180px))` then picks whichever is smaller: on a 900 px-tall window, `100vh - 180px = 720px`, which is less than 760 px, so the image is capped at 720 px. On a 1080 px monitor, `100vh - 180px = 900px`, which exceeds 760 px, so the fixed ceiling wins. The image is never taller than 760 px and never overflows the visible area. The `calc()` from §3 — the same tool used in `top: calc(100% + 10px)` — composes directly into the `min()` argument without any special syntax.

### max() — the dual

`max(A, B)` picks the **larger** of its arguments and sets a floor that can be raised by a second constraint. It is the exact dual of `min()`: `max(200px, 20vh)` means "at least 200 px, but grow with the viewport." The widget leans heavily on `min()` for capping sizes on small screens, but the same family of reasoning applies — wherever you would otherwise write an `@media` query to bump a value up or down at a breakpoint, a `max()` or `min()` call may express the intent directly and continuously.

All four math functions — `calc()`, `clamp()`, `min()`, `max()` — are Baseline Widely Available since 2020 and need no fallback for any browser that also supports the rest of the widget's CSS.

With sizing handled, the next question is how the widget makes its popovers and lightbox appear at all — and that is the job of `@keyframes` and the `animation` shorthand.

## @keyframes & animation

When a popover appears — or a lightbox slides into focus — the event needs a signature. Not decoration for its own sake, but a physical cue: something was absent a moment ago, now it's present. CSS `@keyframes` is the tool for that kind of one-shot entrance: a timeline that plays once when the element mounts and then stops, leaving the element in its final state.

### The keyframe timeline

A `@keyframes` block names a timeline and defines property values at stops along it. `from` and `to` are aliases for `0%` and `100%` of the animation's total duration; between them, the browser interpolates values using the animation's easing function. The `animation` property on a selector references the timeline by name and sets how it plays:

```css
/* styles.css:397 */
animation: popover-in 160ms ease-out;
```

The shorthand order is `name duration easing-function`. These three are all the widget's animations ever specify; fill-mode, delay, iteration count, and play state all fall back to their defaults (no delay, runs once, stops at the initial state after it finishes).

### Popover: the centering-offset subtlety

The popover's keyframe rig is the most instructive of the three:

```css
/* styles.css:400–403 */
@keyframes popover-in {
  from { opacity: 0; transform: translate(-50%, -4px); }
  to   { opacity: 1; transform: translate(-50%, 0); }
}
```

The popover's resting position is established in §3 by `left: 50%; transform: translateX(-50%)` — the centering idiom. When `@keyframes` are active, they *replace* the element's computed `transform` at every frame. If the keyframes animated only the Y axis — `translateY(-4px)` → `translateY(0)` — the `-50%` X offset would be wiped out for the 160ms of animation, snapping the popover to the wrong horizontal position while it fades in. The fix is to carry both axes in the two-argument form of `translate()`: the `-50%` X stays constant across every frame while Y travels from `-4px` to `0`. This is not obvious from reading the shorthand on the element; you have to look at the keyframe to understand why the centering still holds during animation.

### Backdrop: opacity only

```css
/* styles.css:539 */
animation: lb-fade 140ms ease-out;

/* styles.css:548–551 */
@keyframes lb-fade {
  from { opacity: 0; }
  to   { opacity: 1; }
}
```

The backdrop fades in without moving. It doesn't need to travel — its purpose is to materialise and direct attention toward the dialog above it. Adding a slide or scale would introduce noise exactly where the design wants quiet.

### Dialog: scale-and-fade

```css
/* styles.css:564 */
animation: lb-pop 160ms ease-out;

/* styles.css:567–570 */
@keyframes lb-pop {
  from { opacity: 0; transform: scale(0.97); }
  to   { opacity: 1; transform: scale(1); }
}
```

Three percent of scale is below the threshold most people consciously register, but the eye catches the growth. Paired with the fade, the effect reads as the dialog snapping into solidity — a window popping open. More scale (say, 0.85) would feel dramatic; less (0.99) would feel imperceptible. 0.97 is in the range where the motion is felt without being named.

### Why transform and opacity

All three animations touch only `transform` and `opacity`. Modern browsers hand these two properties entirely to the compositor thread, which runs independently of JavaScript and the main rendering pipeline. The element is already painted; the compositor adjusts its position matrix or alpha channel per frame — no layout recalculation, no repaint. Any other animated property (`width`, `background-color`, `top`, `margin`) forces the main thread back into the loop on every frame, making smooth animation fragile on lower-powered devices. `transform` and `opacity` are the two properties that animate without that cost.

### Reduced motion: two halves, two property types

The `@media (prefers-reduced-motion: reduce)` block at the bottom of the stylesheet has two distinct halves. The first (`styles.css:847–857`) targets hover and focus state changes — card lift, button pulse, close-button transitions — and sets `transition: none !important` on them. The second targets the entrance animations:

```css
/* styles.css:858–862 */
.info-popover,
.lb-backdrop,
.lb-dialog {
  animation: none !important;
}
```

[Tutorial 01](./01-function-plotter.md) established the `@media (prefers-reduced-motion: reduce)` mechanics and the default-on / reduce-to-disable pattern. The difference here is which CSS property gets suppressed. In 01, the target was `transition` — ongoing state-change effects on focus. Here, the one-shot entrance animations are `@keyframes`-based, so `animation: none` is the correct disarm. The two properties are independent: `transition: none` has no effect on `@keyframes` animations, and `animation: none` has no effect on `transition`-based ones. The two halves of this block each suppress exactly the motion type they're responsible for.

The `!important` is intentional. Reduced motion is an accessibility preference — a signal from the user, through the OS, that motion should be minimised. An author component rule with higher specificity or a later source position could otherwise outrank the override. `!important` in an accessibility context is one of the few cases the flag exists for: it makes the preference unconditional, regardless of whatever specificity game the component layer is playing.

The elements these animations bring on screen also use `transform` extensively for layout and visual purposes beyond animation — centering, hover lift, rotation, scale states — and that full vocabulary is the subject of the next section.

## Transform and transform-origin

The positioning section used `translate` to solve a geometry problem: `translateX(-50%)` to centre the popover, `translateY(-50%)` to vertically align the lightbox nav buttons. Those were incidental uses. Here, `transform` becomes the topic: a systematic look at every transform function in the widget, how they compose, and how the pivot point changes what they do.

### Hover lift: translate on the card

The thumbnail card rises three pixels when hovered or keyboard-focused:

```css
/* styles.css:275–280 */
.grid-cell--img:hover,
.grid-cell--img:focus-visible {
  transform: translateY(-3px);
  border-color: var(--thumb-border-hover);
  box-shadow: var(--thumb-shadow-hover);
}
```

`translateY(-3px)` moves the element upward in its own coordinate system without touching layout — surrounding cells stay put, the grid does not reflow. Paired with `box-shadow` deepening simultaneously via the `160ms ease` transition declared at `styles.css:271`, the card appears to lift off the page. Three pixels is calibrated to be noticeable but not theatrical.

### Tunnelling hover into a child transform

The image inside the card zooms at the same time, but the rule targeting it lives on the parent's hover state:

```css
/* styles.css:302–305 */
.grid-cell--img:hover .thumb-frame img,
.grid-cell--img:focus-visible .thumb-frame img {
  transform: scale(1.03);
}
```

The descendant combinator (the space between `.grid-cell--img:hover` and `.thumb-frame img`) does the work: CSS evaluates `:hover` on the ancestor and then walks down to style any matching descendants. No JavaScript event listener, no class toggle — the parent's interactive state tunnels directly into the child's transform declaration.

The zoom is contained because `.grid-cell--img` has `overflow: hidden` (`styles.css:268`). Without it, a `scale(1.03)` on a full-width image would visually overflow the card boundary. With it, the image expands inside a clipped frame and the zoom reads as a window opening inward rather than a box growing outward. The two rules — overflow on the container, scale on the descendant — form a unit; neither achieves the intended effect alone.

### Button scale on open state

The info button pulses on hover and holds that scale while the popover is open:

```css
/* styles.css:366–371 */
.info-btn:hover,
.info-btn--open {
  background: var(--accent-mid);
  border-color: var(--accent);
  transform: scale(1.12);
}
```

`.info-btn--open` is a class added by React when the popover is visible, so the scale persists as long as the popover is open — the button stays pressed-looking. `scale()` applies uniformly in both axes from the element's centre (the default `transform-origin`), so the button expands symmetrically without shifting adjacent text.

### Function chaining: the tooltip diamond

The tooltip arrow is a 12×12 px square with borders on its top and left edges:

```css
/* styles.css:414 */
transform: translateX(-50%) rotate(45deg);
```

Two functions, one declaration. They apply left-to-right, each operating in the coordinate frame left by the previous function. `translateX(-50%)` shifts the square left by half its own width — centring it under the popover (the centering idiom from §3). Then `rotate(45deg)` spins it 45° in that already-shifted frame. The result: a centred diamond. The top and left borders, now rotated, become the visible upper-left edges of the arrow pointing up into the popover body.

Order is not decorative — it is geometric. If you reversed the functions to `rotate(45deg) translateX(-50%)`, the element would rotate first, and the subsequent translation would move it 50% of its width *along the rotated axis* (now diagonal), landing it in the wrong position entirely. Each function changes the local coordinate system for the next one.

### transform-origin: where the pivot lives

The fit-to-viewport wrapper has a single declaration:

```css
/* styles.css:107–109 */
.fit-scale {
  transform-origin: top center;
}
```

No `transform` value appears in the CSS at all. The scale itself is injected by `App.tsx` as an inline style: a `ResizeObserver` measures the available viewport height, computes a scale factor `s`, and writes `transform: scale(s)` directly onto the element's `style` attribute. CSS owns the pivot; JS owns the magnitude.

The pivot matters because the default `transform-origin` is `50% 50%` — the element's centre. With the default, `scale(0.85)` would shrink the widget inward from all four sides simultaneously, causing it to float away from the top of its container. `top center` fixes the anchor at the top edge so the widget shrinks downward. The top stays in place; only the bottom moves. The frontend tutorial (`../frontend/02-image-comparison-table.md`) covers how `App.tsx` compensates with a negative `margin-bottom` to reclaim the visual space the scale removes.

### Recap: translateY on the nav buttons

The lightbox nav buttons (`styles.css:598`) use `transform: translateY(-50%)` with `top: 50%` — the classic absolute-centering trick introduced in §3. It appears here again as a reminder that `translate` is the tool CSS reaches for whenever vertical centering needs to be self-sizing: unlike a negative margin, `translateY(-50%)` uses the element's own height as the reference, so it works regardless of button height.

### The side effects every transform brings

Any element with `transform` other than `none` creates a new stacking context (as detailed in §3) and becomes a containing block for any `position: fixed` descendants inside it. That second effect is surprising: a `position: fixed` child that you expect to be anchored to the viewport will instead be anchored to the transformed ancestor.

The `.lb-nav` buttons are deliberately `position: absolute`, not `position: fixed` (`styles.css:596`). They position themselves relative to the lightbox dialog, which is what the layout requires. Had they been `fixed`, the transform on an ancestor would have broken that positioning. Choosing `absolute` here sidesteps the containing-block trap entirely — the transform's side effect becomes irrelevant.

The visual surface of the widget is nearly complete. The next layer addresses the glass-like quality of the lightbox backdrop, the square thumbnail slots, and how images fill them — `backdrop-filter`, `aspect-ratio`, and `object-fit`.

## Frosted Glass, Square Thumbnails, and Fitting Images

Three CSS features work together throughout the widget to handle image presentation: `backdrop-filter` puts frosted-glass panels over content, `aspect-ratio` makes thumbnails self-sizing squares, and `object-fit` decides whether an image crops to fill its box or letterboxes to show everything. Each one has a specific mechanic that has to be understood before it can be applied correctly.

### backdrop-filter: blurring what is behind you

`backdrop-filter` applies a filter to the area painted *behind* the element — not to the element's own pixels. That distinction is everything. The property is written on the element you want to look frosted, but the blurring happens to whatever shows through it.

Because the filter only affects the backdrop, the element must be at least partially transparent for the effect to be visible at all. A fully opaque background covers the blurred layer completely. All three uses in this stylesheet pair `backdrop-filter` with a semi-opaque `rgba` background.

The zoom badge on each thumbnail:

```css
/* styles.css:317–320 */
background: rgba(15, 23, 42, 0.65);
color: #f8fafc;
-webkit-backdrop-filter: blur(4px);
backdrop-filter: blur(4px);
```

The slate `rgba` at 65% opacity lets the thumbnail show through at 35%, and the blur softens it. Without the `rgba` background the badge would be fully transparent and the blur would have nothing visible to show through.

The lightbox overlay uses the same technique at the viewport scale:

```css
/* styles.css:532–533 */
-webkit-backdrop-filter: blur(4px);
backdrop-filter: blur(4px);
```

And the navigation buttons frosted over the displayed image:

```css
/* styles.css:611–612 */
-webkit-backdrop-filter: blur(6px);
backdrop-filter: blur(6px);
```

The nav buttons use a slightly lighter background (55% opacity instead of 65%) because they sit over the already-darkened lightbox image rather than over bright thumbnail content.

**Browser compatibility.** Older Safari — versions before Safari 18 — requires `-webkit-backdrop-filter`; Chrome and Firefox have shipped the unprefixed form since 2019 and 2022 respectively, and Safari 18 (2024) added unprefixed support. This stylesheet now pairs both declarations, with the prefixed form first:

```css
-webkit-backdrop-filter: blur(4px);
backdrop-filter: blur(4px);
```

The prefixed-first ordering is the established convention for vendor-prefix fallbacks: engines that only understand `-webkit-backdrop-filter` (older Safari < 18) apply the first declaration and ignore the unknown second; engines that support both use the unprefixed declaration because it appears later in the cascade. Note also that `backdrop-filter` creates a new stacking context (as in §3) — any sibling that needs to stack above a backdrop-filter element must account for that isolation.

### aspect-ratio: deriving height from width

Before `aspect-ratio`, a common technique for enforcing a box's proportions was to set `padding-top` to a percentage and use absolute positioning inside — a hack that worked because percentage padding is relative to the element's *width*. `aspect-ratio` does the job directly.

```css
/* styles.css:287–292 */
.thumb-frame {
  display: block;
  aspect-ratio: 1 / 1;
  width: 100%;
  background: var(--thumb-bg);
}
```

`width: 100%` stretches the frame to fill its grid cell. With both dimensions unconstrained, `aspect-ratio: 1 / 1` kicks in: the browser derives the height from the width, producing a square that fits whatever width the grid cell provides. No fixed heights, no breakpoints.

The same ratio on the missing-cell placeholder:

```css
/* styles.css:332–337 */
.grid-cell--missing {
  background: var(--thumb-bg);
  border: 1.5px dashed var(--thumb-border);
  border-radius: 14px;
  aspect-ratio: 1 / 1;
}
```

The placeholder occupies the same space as a real thumbnail, so the grid row does not collapse when an image is absent.

In compact mode, the per-image aspect ratio widens to 4:3:

```css
/* styles.css:515 */
.grid-wrap--compact .thumb-frame img { aspect-ratio: 4 / 3; height: 100%; }
```

A 4:3 frame is a better fit for most photographs than a square crop, and compact mode shows only two columns where each image has more horizontal room to breathe.

The picker thumbnails follow the same square pattern:

```css
/* styles.css:803–807 */
.pick-thumb {
  display: block;
  aspect-ratio: 1 / 1;
  overflow: hidden;
}
```

### object-fit: cover vs contain

`object-fit` controls how a replaced element — an `<img>` — fills the box it sits in. For it to do anything, the image must have explicit dimensions; without them, there is no box to fit within. Every thumbnail image in this stylesheet gets:

```css
/* styles.css:294–300 */
.thumb-frame img {
  display: block;
  width: 100%;
  height: 100%;
  object-fit: cover;
  transition: transform 240ms ease;
}
```

`width: 100%; height: 100%` stretches the `<img>` element to fill the `aspect-ratio`-sized `.thumb-frame` box entirely. `object-fit: cover` then scales the image up until it fills that box completely — cropping the parts that overflow while keeping the image's own proportions intact. For a grid of thumbnails this is exactly right: every cell is a clean filled square regardless of whether the source image is portrait, landscape, or square.

The picker repeats the pattern:

```css
/* styles.css:808–813 */
.pick-thumb img {
  display: block;
  width: 100%;
  height: 100%;
  object-fit: cover;
}
```

The lightbox makes the opposite choice:

```css
/* styles.css:583–593 */
.lb-image {
  display: block;
  max-width: 100%;
  max-height: min(760px, calc(100vh - 180px));
  object-fit: contain;
  width: auto;
  height: auto;
}
```

`object-fit: contain` scales the image down until it fits entirely within its box, letterboxing any leftover space. In the thumbnail grid, cropping serves visual density — each cell is full and uniform. In the lightbox, the user opened the image to inspect it, so cropping would hide the very content they want to see. `contain` shows the whole image, and the `max-height` constraint (from §4's `min()`) keeps it from overflowing the viewport.

The choice between `cover` and `contain` maps directly to intent: crop for grid density, show all for detail.

With the image and glass techniques in place, the next section turns to `::before` and `::after` pseudo-elements — the mechanism the stylesheet uses to inject decorative and text content without touching the markup.

## Pseudo-element Content & the `quotes` Property

Two small typographic details give the widget its polish: a thin coloured tick beside every row label, and curly quotation marks around every model prompt. Neither requires a DOM element. Both come from CSS generated content — either a `::before`/`::after` pseudo-element, or the browser's built-in quoting behaviour on `<q>` controlled by the `quotes` property.

### The `content` requirement

`::before` and `::after` generate a child box at the start or end of an element's content, but they only generate a box at all if a `content` declaration is present. The default is `content: none`, which suppresses box generation entirely. To create a pseudo-element that has no text but exists as a physical box in the layout, use `content: ''` — an empty string. That is the idiom for decoration-only pseudo-elements: the empty string satisfies the requirement so a box is generated; everything visible is then done by other properties.

The row-label accent bar is exactly this:

```css
/* styles.css:241–251 */
.grid-cell--label::before {
  content: '';
  display: block;
  width: 3px;
  align-self: stretch;
  min-height: 18px;
  border-radius: 2px;
  background: var(--accent);
  flex-shrink: 0;
  opacity: 0.7;
}
```

`content: ''` brings the box into existence. After that, the declaration does nothing else visual — the bar's appearance is entirely the work of `width: 3px`, `background: var(--accent)`, and the rounding from `border-radius: 2px`.

The layout context matters here. `.grid-cell--label` is a flex container, so the `::before` pseudo-element participates as a flex item. `align-self: stretch` makes it fill the full cross-axis height of the cell — whatever height the label text occupies, the accent bar matches it. `min-height: 18px` gives it a floor in case the text is very short, and `flex-shrink: 0` prevents it from narrowing under pressure. [Custom properties](./01-function-plotter.md) inherit into pseudo-elements the same way they inherit into regular elements, so `var(--accent)` resolves against the cascade normally.

In compact mode the accent bar disappears:

```css
/* styles.css:487 */
.grid-wrap--compact .grid-cell--label::before { display: none; }
```

`display: none` suppresses the box entirely, removing the pseudo-element from the flex formatting context so it contributes no space at all.

### `attr()` — reading attribute values into `content`

Alongside the accent bar suppression, the compact-mode block adds a live column-label badge — a `::after` pseudo-element that floats a chip over each thumbnail:

```css
/* styles.css:490–512 */
.grid-wrap--compact .grid-cell--img::after {
  content: attr(data-label);
  position: absolute;
  left: 8px;
  bottom: 8px;
  max-width: calc(100% - 16px);
  overflow: hidden;
  text-overflow: ellipsis;
  white-space: nowrap;
  font-size: 11px;
  font-weight: 600;
  line-height: 1.2;
  color: #f8fafc;
  background: rgba(15, 23, 42, 0.65);
  -webkit-backdrop-filter: blur(4px);
  backdrop-filter: blur(4px);
  border-radius: 7px;
  padding: 3px 7px;
  pointer-events: none;
}
.grid-wrap--compact .grid-cell--img {
  position: relative;
}
```

`attr(name)` is a CSS function that reads the value of the named HTML attribute on the matched element and returns it as a string for use in `content`. It has been part of CSS since CSS2 and is broadly supported in all browsers for the `content` property. `Grid.tsx:57` sets `data-label={col.label}` on every `.grid-cell--img` button, so `attr(data-label)` resolves to the column name — "DALL·E 3", "Stable Diffusion", etc. — and the `::after` box renders that string as a visible chip.

The badge exists because compact mode hides the header row (Step 3), removing the model names from view. Scoping to `.grid-wrap--compact` keeps it invisible in normal mode, where the header row already carries the labels.

**Positioning within the button.** `.grid-wrap--compact .grid-cell--img { position: relative; }` establishes the button as a containing block for the absolutely positioned pseudo-element. `position: absolute; left: 8px; bottom: 8px` pins the chip to the thumbnail's bottom-left corner.

**Legibility treatment.** A translucent-dark `rgba(15, 23, 42, 0.65)` background with light `#f8fafc` text keeps the chip readable over any image. `-webkit-backdrop-filter: blur(4px); backdrop-filter: blur(4px)` (the prefixed form targets older Safari < 18; see §7) adds the same frosted-glass effect used on the zoom badge and the lightbox. `max-width: calc(100% - 16px); overflow: hidden; text-overflow: ellipsis; white-space: nowrap` prevent a long model name from escaping the cell boundary. `pointer-events: none` ensures the chip never intercepts a click intended for the button beneath it.

### `quotes` and the `<q>` element

The model prompts shown in the info popover and the lightbox are marked up as `<q>` elements:

- `InfoPopover.tsx:81` — `<q class="prompt-text">{prompt}</q>`
- `Lightbox.tsx:168` — `<q class="lb-prompt">{row.prompt}</q>`

`<q>` is the HTML inline quotation element. Browsers automatically generate opening and closing quotation marks on its `::before` and `::after` pseudo-elements — this happens via the browser's default stylesheet, not the author's. The characters used are controlled by the CSS `quotes` property.

```css
/* styles.css:456 */
.prompt-text { quotes: "\201C" "\201D"; }

/* styles.css:711 */
.lb-prompt  { quotes: "\201C" "\201D"; }
```

`\201C` and `\201D` are the Unicode code points for left and right double quotation marks — the curly `"…"` pair. Without this rule, browsers fall back to locale-dependent defaults, which may be straight ASCII `"` or language-specific guillemets. With it, every prompt renders with typographic curly quotes regardless of the operating system's locale.

The browser's default stylesheet inserts `open-quote` and `close-quote` as `content` values on `<q>::before` and `<q>::after`. Those keywords are resolved against the `quotes` property. Setting `quotes` on `.prompt-text` or `.lb-prompt` controls which characters the browser-inserted pseudo-elements produce — with no `::before`/`::after` rules required in the author stylesheet at all.

### A real element doing a pseudo-element's job

One tooltip-arrow shape in this widget does *not* use a pseudo-element. The `<span class="info-arrow" aria-hidden="true" />` in `InfoPopover.tsx` achieves its rotated-square diamond shape via `position: absolute; transform: translateX(-50%) rotate(45deg)` on a real DOM node (see `styles.css:405–415`). The same visual is commonly done with `::before` or `::after` on the tooltip container — one less DOM element — but a real element works just as well, and in this case the explicit `aria-hidden` attribute makes the decorative intent clear.

With generated content and quotation marks handled, the remaining interaction-layer techniques — `:focus-visible` focus rings, `appearance: none` resets, and the rest of the keyboard-polish declarations — are next.

## :focus-visible, appearance: none & Interaction Polish

Every interactive element in this widget — thumbnail buttons, image picker cards, lightbox nav arrows, the popover close button — needs a focus indicator keyboard users can track. The challenge is doing that without showing a ring every time a mouse user clicks. That is what `:focus-visible` solves, and it underpins the interaction layer of this stylesheet.

### :focus vs :focus-visible

[Tutorial 01](./01-function-plotter.md) introduced `:focus` for the text input — a pseudo-class that matches whenever an element has focus, regardless of how that focus arrived. For a text field that is correct: any focus means the user is about to type. For a button styled as an image tile it is noisy: a mouse click already provides visual feedback through the hover state, and a focus ring that pops up then vanishes on pointer-move is jarring.

`:focus-visible` uses a browser heuristic instead. If the user is navigating by keyboard (Tab, arrow keys), `:focus-visible` matches. If the user clicked or tapped a non-input element, it does not. Keyboard users get rings; mouse users do not:

```css
/* styles.css:282–285 */
.grid-cell--img:focus-visible {
  outline: 2px solid var(--accent);
  outline-offset: 2px;
}
```

The same pattern appears on `.info-btn` (`styles.css:373–376`), `.info-popover-close` (`styles.css:443`), `.lb-nav` (`styles.css:616`), `.lb-close` (`styles.css:686`), and `.pick-card` (`styles.css:795`) — every interactive element in the widget.

### The :hover / :focus-visible pairing idiom

There is a subtler requirement beyond showing a ring: keyboard users should receive the same affordances that mouse users get on hover. The widget handles this with a two-rule structure. Consider `.grid-cell--img`:

```css
/* styles.css:275–280 */
.grid-cell--img:hover,
.grid-cell--img:focus-visible {
  transform: translateY(-3px);
  border-color: var(--thumb-border-hover);
  box-shadow: var(--thumb-shadow-hover);
}

/* styles.css:282–285 */
.grid-cell--img:focus-visible {
  outline: 2px solid var(--accent);
  outline-offset: 2px;
}
```

Rule one is a comma-separated selector list: hover and keyboard focus both trigger the lift — the `-3px` translate, the brightened border, the elevated shadow. A keyboard user tabs to a thumbnail and it rises exactly as it would under the pointer. Rule two is keyboard-only: the `outline` ring appears on top of the lifted state, adding an unambiguous signal that focus arrived by keyboard. Mouse users see the lift but never the ring.

`.pick-card` is an `<a>` element in the image picker, and it follows the identical two-rule structure at `styles.css:789–795`. Shared affordance first; explicit ring second.

### appearance: none

Each thumbnail button is a `<button>` element. Without intervention the browser renders it with its native chrome — system bevel, platform-specific background, OS-themed text colour. `appearance: none` at `styles.css:263` clears all of that:

```css
/* styles.css:263 */
appearance: none;
```

After this reset, only the stylesheet's own declarations contribute to the visual. The same reset appears on `.info-popover-close` (`styles.css:432`), `.lb-nav` (`styles.css:595`), and `.lb-close` (`styles.css:675`) — every element the browser would otherwise render as a system widget. `outline: none` at `styles.css:272` then discards the default focus ring on `.grid-cell--img`; the `:focus-visible` block above restores a deliberate replacement, following the same contract [tutorial 01](./01-function-plotter.md) established for the text input.

### outline-offset and the explicit ring

[Tutorial 01](./01-function-plotter.md) used `box-shadow: 0 0 0 3px` as a focus ring because it follows `border-radius`. Here the ring uses `outline` with `outline-offset`:

```css
/* styles.css:283–285 */
outline: 2px solid var(--accent);
outline-offset: 2px;
```

`outline-offset` pushes the ring outside the element's border by the given distance. On an element with its own 1.5px border (`styles.css:264`), a flush outline and the border can merge visually. A 2px gap separates them cleanly. The `.lb-nav:focus-visible` rule uses `outline: 2px solid #f8fafc` (`styles.css:616`) — white rather than accent — because the nav buttons float over a dark image where the accent blue blends in.

### :disabled

The lightbox nav arrows are `<button>` elements that `Lightbox.tsx` marks `disabled` when there is no further row or column to navigate to:

```css
/* styles.css:617 */
.lb-nav:disabled { opacity: 0.25; cursor: not-allowed; }
```

`opacity: 0.25` dims the button to a quarter of its normal visibility. `cursor: not-allowed` shows the blocked-circle pointer before a click is attempted. Specificity: `.lb-nav:disabled` is (0,2,0) — one class plus one pseudo-class — against `.lb-nav` at (0,1,0), so the disabled rule overrides the base without needing `!important`.

### pointer-events: none

The `.thumb-zoom` badge is absolutely positioned over the thumbnail button. A click landing on the badge would be consumed by the badge rather than the button behind it:

```css
/* styles.css:323 */
pointer-events: none;
```

The badge becomes visually present but click-through. All pointer events pass to the `.grid-cell--img` button underneath, which opens the lightbox.

### Minor polish

Three small techniques finish the interaction layer. `border-radius: 999px` at `styles.css:168`, `226`, and `671` creates a pill on the footnote hint, the column tag, and the lightbox tag. As [tutorial 01](./01-function-plotter.md) noted, `border-radius` clips to half the element's shortest side, so 999 px — safely above any realistic dimension — always yields a capsule without knowing the element's width.

`font-variant-numeric: tabular-nums` at `styles.css:753` on `.lb-pos` requests equal-width digit glyphs. The position counter displays "row 3/8 · col 2/5"; without tabular digits, single-digit values swapping for double-digit ones reflow the layout on each navigation step. Tabular digits eliminate the jitter.

Finally, `.card` carries `border-top: 2.5px solid var(--accent)` at `styles.css:116` — 1.5px thicker than the card's 1px border — producing a visible accent stripe at the top edge. The same `--accent` token colours the focus rings, the label bar from §8, and the eyebrow text: one token, three scales of emphasis.
